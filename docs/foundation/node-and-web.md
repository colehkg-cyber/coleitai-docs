# 3강. 노드 & 웹의 작동 원리

> **웹사이트 = 건물. 프론트엔드 = 외관, 백엔드 = 배관, 서버 = 부지.** 이 구조를 알면 Lighthouse 400점이 왜 중요한지 이해된다.

---

## 웹은 건물이다

| 건물 | 웹 | 역할 |
|------|-----|------|
| 외관/인테리어 | **프론트엔드** (HTML, CSS, JS) | 사용자 눈에 보이는 것 |
| 배관/전기/엘리베이터 | **백엔드** (서버 로직, DB) | 보이지 않지만 작동하는 것 |
| 부지/토지 | **서버/호스팅** (Vercel, AWS) | 건물이 서 있는 땅 |
| 건물 주소 | **도메인** (myblog.com) | 사람들이 찾아오는 주소 |
| 건축법 | **웹 표준** (W3C) | 이 규칙대로 지어야 안전 |

## Node.js — JavaScript를 서버에서도

### JavaScript의 원래 역할
원래 JavaScript는 **브라우저 안에서만** 작동하는 언어였습니다.
→ 버튼 클릭, 애니메이션, 폼 검증 등 "화면"에서만 사용.

### Node.js가 바꾼 것
2009년 Ryan Dahl이 만든 Node.js는 JavaScript를 **서버에서도 쓸 수 있게** 했습니다.

```
Before Node.js:
  프론트엔드 → JavaScript
  백엔드 → Python, Java, Ruby, PHP...
  = 2개 언어 필요

After Node.js:
  프론트엔드 → JavaScript
  백엔드 → JavaScript (Node.js)
  = 1개 언어로 전부!
```

:::tip 💡 왜 중요한가?
여러분이 쓸 블로그 템플릿(Next.js)이 **Node.js 기반**입니다.
"Node.js가 뭔지" 알면 에러가 나도 겁나지 않습니다.
:::

## 클라이언트-서버 모델

```
[브라우저/앱]  ←→  [인터넷]  ←→  [서버]
 (클라이언트)                      (데이터 저장소)
   요청 보냄 →                  ← 응답 보냄
```

### 브라우저에 URL을 치면 일어나는 일

1. **DNS 조회** — "myblog.com"의 실제 IP 주소 찾기 (전화번호부)
2. **TCP 연결** — 서버와 통신 채널 열기 (전화 걸기)
3. **HTTP 요청** — "이 페이지 주세요" (주문하기)
4. **서버 처리** — 요청된 페이지 준비 (요리하기)
5. **HTTP 응답** — HTML/CSS/JS 전송 (음식 나옴)
6. **렌더링** — 브라우저가 화면에 그림 (먹기)

이 전체 과정이 **0.5~3초** 안에 일어납니다.

## SSG vs SSR vs CSR

블로그에 가장 중요한 개념입니다:

### SSG (Static Site Generation) — 미리 만들어두기 ⭐
```
빌드 시점에 모든 페이지를 HTML로 미리 만들어둠
→ 요청이 오면 만들어둔 HTML을 바로 전송
→ 가장 빠름! CDN에 캐시 가능
```
**비유:** 도시락을 미리 만들어두고, 주문이 오면 바로 건네줌.

### SSR (Server-Side Rendering) — 주문 들어오면 만들기
```
요청이 올 때마다 서버에서 HTML을 실시간 생성
→ SSG보다 느림, 서버 부하 있음
→ 실시간 데이터가 필요한 경우 사용
```
**비유:** 주문이 올 때마다 즉석에서 요리.

### CSR (Client-Side Rendering) — 브라우저가 만들기
```
빈 HTML + JavaScript를 보냄
→ 브라우저에서 JavaScript가 화면을 그림
→ 초기 로딩 느림, SEO 불리
```
**비유:** 재료를 보내고 손님이 직접 요리.

| | SSG | SSR | CSR |
|---|---|---|---|
| 속도 | ⚡ 가장 빠름 | 보통 | 초기 느림 |
| SEO | ✅ 최고 | ✅ 좋음 | ❌ 불리 |
| 서버 비용 | 거의 없음 | 높음 | 없음 |
| 블로그 적합도 | ⭐ 최적 | 가능 | ❌ 부적합 |

:::tip 🎯 핵심
**우리 블로그는 SSG를 씁니다.** 이게 Lighthouse 400점의 기술적 비밀입니다.
미리 만들어둔 정적 HTML = 가장 빠름 = 구글이 가장 좋아함.
:::

## Vercel & CDN — 세계 어디서든 빠르게

### CDN (Content Delivery Network)
```
서울에서 미국 서버까지: 200ms (느림)
서울에서 서울 CDN까지: 10ms (빠름!)

CDN = 전 세계에 복사본을 미리 배포
→ 어디서 접속해도 가까운 곳에서 전송
```

### Vercel
- Next.js를 만든 회사
- **자동 CDN 배포** — 코드를 올리면 전 세계에 자동 배포
- **무료 플랜** — 개인 블로그는 충분
- **자동 HTTPS** — 보안 인증서 자동 설정

---

## Lighthouse 400점의 기술적 이유

| 항목 | 점수 | SSG가 좋은 이유 |
|------|------|----------------|
| Performance | 100 | 정적 HTML → 로딩 속도 최고 |
| Accessibility | 100 | 시맨틱 HTML → 접근성 자동 확보 |
| Best Practices | 100 | HTTPS + 최신 표준 자동 적용 |
| SEO | 100 | 완성된 HTML → 구글 크롤러가 바로 읽음 |

```
SSG + Vercel + CDN = Lighthouse 400점
= 구글이 가장 좋아하는 사이트
= 검색 순위 상위
= 트래픽
= 돈
```
